# PentAGI

## Overview

PentAGI is an **autonomous penetration testing agent swarm** — a containerized platform that decomposes security assessment objectives into subtasks, delegates them across a coordinated set of specialized agents, and synthesizes findings into actionable reports. Unlike general-purpose agent frameworks, every design decision is oriented around offensive security workflows: tool sandboxing, network isolation, scope enforcement, and vulnerability knowledge management are first-class concerns.

- **Domain**: Autonomous penetration testing and security research
- **Status**: Active — published February 2026, immediately trended on GitHub; briefly restricted for a license compliance audit before reopening
- **License**: MIT for the core framework; **AGPL-3.0** for the Cloud/hosted SDK component
- **Language**: Go (backend/orchestration) + Python (agent logic and tooling)

## Architecture & Core Concepts

PentAGI is a **containerized microservices platform** organized in three layers:

```
User / GraphQL / REST API
        ↓
  PrimaryAgent (orchestrator)
  Generator → Refiner → Plan
        ↓
┌───────────────────────────────────────┐
│ Research    │ Development │ Execution  │
│ Searcher    │ Coder       │ Pentester  │
│ Enricher    │ Installer   │            │
│ Adviser     │ Reflector   │            │
└───────────────────────────────────────┘
        ↓               ↓
  pgvector (vector)  Neo4j/Graphiti (knowledge graph)
        └────── PostgreSQL ──────┘
```

**Infrastructure**: Docker-in-Docker with five isolated networks. Supports optional two-node deployment splitting API responsibilities from worker execution, allowing the attack surface of the control plane to be separated from live testing activity.

### Agent Roster (13 agents)

| Category | Agent | Role |
|----------|-------|------|
| Orchestration | `PrimaryAgent` | Top-level task decomposition and sub-agent delegation |
| | `Generator` | Breaks high-level objectives into ordered subtasks |
| | `Refiner` | Rewrites and tightens task definitions before dispatch |
| User-facing | `Assistant` | Interactive chat interface for operator guidance |
| Advisory | `Adviser` | Recommends tool strategies and attack paths |
| | `Reflector` | Self-validation layer; corrects agents that drift from structured tool calls toward ungrounded prose |
| Research | `Searcher` | OSINT and web-based reconnaissance queries |
| | `Enricher` | Contextual enrichment of discovered data (IP geo, WHOIS, CVE lookup) |
| Development | `Coder` | Writes and debugs exploit scripts, PoC code |
| | `Installer` | Manages tool installation inside the sandbox container |
| Execution | `Pentester` | Executes security tools and interprets raw output |
| Reporting | `Reporter` | Aggregates findings into structured vulnerability reports |
| Quality | `Validator` | Verifies finding reproducibility and eliminates false positives |

> **Note**: The exact names for the last two roles (Reporter, Validator) are inferred from the architecture description; the published source briefly restricted public access during a license audit and these names may differ in the canonical codebase.

## Agent Communication Model

PentAGI uses a **hierarchical delegation model** rather than a peer-to-peer mesh:

1. The `PrimaryAgent` receives a target and scope from the operator.
2. `Generator` produces a plan (ordered list of subtasks); `Refiner` validates and tightens it.
3. Each subtask is dispatched to the appropriate specialist agent.
4. Specialist agents report structured findings back up the hierarchy.
5. `Reflector` monitors agent outputs in flight and re-routes agents that produce natural-language summaries instead of calling the expected tools — enforcing that execution is deterministic and auditable.

There is no shared message bus between peer agents. Communication flows through the orchestrator, which preserves a clean audit log and makes scope enforcement straightforward (the orchestrator is the only entity that decides which agent acts next).

## Tool & Capability System

PentAGI integrates a broad suite of security tools, all executed inside isolated Docker containers:

- **Network scanning**: nmap, masscan
- **Web application testing**: Burp Suite integration, ffuf, nuclei
- **Exploitation frameworks**: Metasploit, custom PoC scripts generated by `Coder`
- **OSINT**: Shodan, Censys, theHarvester (via `Searcher` and `Enricher`)
- **Password/credential testing**: Hydra, Hashcat (sandboxed)

**Sandboxing model**: Each tool invocation runs inside a fresh container, preventing lateral movement from a compromised tool process into the control plane. Five isolated Docker networks segregate: (1) the control plane, (2) target-facing traffic, (3) the knowledge store, (4) inter-agent messaging, and (5) external egress.

**Scope enforcement**: Target IP ranges and domains are declared at session creation. The `PrimaryAgent` validates every tool invocation against the declared scope before dispatch — out-of-scope calls are blocked and logged.

## Memory & Knowledge Management

PentAGI maintains two parallel knowledge stores, each serving a different retrieval pattern:

| Store | Technology | Purpose |
|-------|-----------|---------|
| Vector store | **pgvector** (PostgreSQL extension) | Semantic similarity search over discovered text, tool outputs, CVE descriptions |
| Knowledge graph | **Neo4j + Graphiti** | Structured relationships: host ↔ service ↔ vulnerability ↔ credential ↔ exploit path |

Findings discovered by one agent (e.g., an open port by `Pentester`) are immediately persisted to both stores, making them available to other agents in the same session without redundant tool calls. The knowledge graph enables path-finding queries like "show all hosts reachable from the initial foothold that have an unpatched CVE."

Report generation pulls from both stores: the `Reporter` agent queries the graph for structured finding metadata and the vector store for supporting evidence, then renders a structured vulnerability report.

## Strengths

- **Domain-specific design**: Every layer — agent roles, memory architecture, network isolation — is purpose-built for penetration testing rather than adapted from a general framework.
- **Dual-store knowledge management**: Combining a vector store (semantic search) with a knowledge graph (structural relationships) is well-suited to the complex entity relationships in security assessments.
- **Scope enforcement at the orchestrator**: Centralizing scope checks in the `PrimaryAgent` provides a single, auditable enforcement point rather than relying on each tool or agent to self-police.
- **Reflector agent**: The self-validation layer that catches agents drifting into prose is a practical solution to a real reliability problem in LLM tool-use pipelines.
- **Sandboxed execution**: Docker-in-Docker with network segmentation meaningfully reduces the blast radius of a misbehaving tool or agent.

---

## Weaknesses

- **Very new**: Published February 2026; limited production usage data, immature ecosystem, unclear long-term maintenance commitment.
- **License ambiguity**: MIT/AGPL-3.0 dual licensing requires careful evaluation for commercial or proprietary engagements. The brief source restriction for a compliance audit suggests licensing was not settled before public release.
- **Black-box orchestration**: The hierarchical delegation model makes debugging a failed subtask harder — it is not always clear which agent in the chain caused a failure or why the `Refiner` rewrote a task in a way that changed its meaning.
- **Limited documentation at time of research**: The source restriction period and early community age mean that architectural details (e.g., exact agent names, inter-agent protocol) are partially inferred rather than confirmed from canonical docs.
- **Specialized scope**: The tight security focus that is a strength for pentesting engagements makes the framework unsuitable for general-purpose agent workflows.

## Comparison with acteon-swarm

| Dimension | PentAGI | acteon-swarm |
|-----------|---------|-------------|
| **Metaphor** | Autonomous red team | General-purpose agent graph |
| **Domain** | Penetration testing only | Domain-agnostic |
| **Orchestration** | Hierarchical; single active orchestrator | Flexible topology (peer, hierarchical, or hybrid) |
| **State / Memory** | Dual-store (pgvector + Neo4j) per session | Configurable; memory is a first-class concern but not prescribed |
| **Parallelism** | Sequential plan execution (subtasks ordered) | Native concurrent agent dispatch |
| **Tool sandboxing** | Docker-in-Docker; mandatory network isolation | User-defined; framework-agnostic |
| **Scope enforcement** | Built-in at orchestrator layer | Application responsibility |
| **Use case fit** | Authorized security assessments, red teaming | General multi-agent automation, including security workflows |

**Key tradeoff**: PentAGI trades generality for depth. It ships with security-domain knowledge baked into agent roles, tool integrations, and memory schema, eliminating significant upfront setup for pentesting work. acteon-swarm requires the operator to bring that domain knowledge but scales to any workflow without framework-level constraints.

For a security team running authorized assessments, PentAGI's integrated toolchain and scope enforcement are genuinely valuable. For a team building a mixed-use automation platform that includes some security workflows, acteon-swarm's flexibility avoids coupling the entire stack to a single-domain framework.
